### 에이전트 강화 논의

- 스톡 대시보드의 본질
    - 단순 정보가 아닌, 판단의 근거가 되는 정보 제공.
    - 따라서 판단 근거가 되는 정보가 무엇인지 정의 (또는 여러가지중 선택)한후 정해진 답변 틀에 맞추어서 **정보 큐레이팅**
- 현재 Workflow
    - 지표 추출 (raw 데이터 계산) → 시그널 추출 → 정규식 설명 → llm 답변 (무형식)
- 이상 Workflow
    - 사용자 질문 → 1차 라우팅 (페이지별) → 2차 라우팅 (페이지내) → 정보 큐레이팅 → Open to Feedback
    - 1차 라우팅: 질문에 상응하는 페이지 제시 → 에이전트가 직접 이동까지
    - 2차 라우팅: 먼저 현 페이지 설명 (템플릿) → 질문에 상응하는 아이템 제시 → 아이템 결과 반환
    - 큐레이팅: 아이템 결과를 종합하여 시각화 + 답변 제공
    - 피드백 대기: 사용자의 추가 질문에 대한 대응
- 필요 에이전트 리스트
    - 페이지 라우터: page_explanation.md
    - 아이템 라우터: 4종류 (각 페이지별)
        - 가격(단일) : 가격만 있으면 라우터는 필요 없을 수 있음
        - 상관(상관, Gapfill): 히트맵, 산점도/스프레드 각각 라우팅
        - 지표: RSI, MACD, ATR+vol 각각 라우팅
        - 전략: 모멘텀, 역추세, 위험회피 라우팅
    - 리포트 오거나이저: 1,2차 라우터가 정해준 그 아이템 반환값 + 그 분야 지식 엑스퍼트 호출 (프롬프트 + output structure)
    - 지식 엑스퍼트: Pydantic 모델 4종류
        - 가격, 상관, 지표, 전략
    - 토털 오케스트레이터: 페이지별/아이템별/그리고 프로젝트 컨텍스트 이외의 것 처리. 필요한지 검토 요

# project-wrapup 문서 개편 가이드

## 목적
`project-wrapup` 문서는 단순 회고가 아니라, 유사 프로젝트가 그대로 참고할 수 있는 실행 가이드여야 한다.
핵심 목적은 다음 세 가지다.

1. 반복된 시행착오를 예방하는 규칙을 남긴다.
2. 유사 프로젝트가 바로 가져다 쓸 수 있는 전문성 있는 산출물을 제공한다.
3. 불필요한 Phase 증식과 trial-and-error를 줄이는 이상적 개발 순서를 제안한다.

## 문서 전체 원칙

- 회고 중심 문장보다 지침 중심 문장을 우선한다.
- "무슨 일이 있었는가"보다 "다음 프로젝트는 어떻게 해야 하는가"를 더 분명히 쓴다.
- 모든 중요한 교훈은 가능한 한 예방 규칙, 사전 점검, 재사용 패턴과 연결한다.
- 문서 간 역할이 겹치지 않게 분리한다.
- 유사 프로젝트의 실무자가 바로 읽고 적용할 수 있게 작성한다.

## 1. project-blueprint.md 작성 가이드

### 역할
`project-blueprint.md`는 단순한 Phase 목록이 아니라,
유사 프로젝트를 위한 이상적 개발 계획서여야 한다.

### 반드시 담아야 할 내용
- 무엇을 먼저 만들고 무엇을 나중에 만들어야 하는지
- 무엇은 반드시 선행돼야 하고 무엇은 미뤄도 되는지
- 어떤 설계 결정을 초기에 고정해야 하는지
- 어떤 고급 기능은 언제까지 시작하지 말아야 하는지
- trial-and-error를 줄이기 위한 권장 순서

### 권장 구조
- 프로젝트 유형 요약
- 권장 개발 경로
- Core Path
- Expansion Path
- Advanced Path
- 각 단계별:
  - 목표
  - 선행조건
  - 완료조건
  - 반드시 초기에 결정할 사항
  - 아직 시작하지 말아야 하는 조건
- 불필요한 Phase 증식 방지 규칙
- 추천 순서의 근거
- 유사 프로젝트용 일정 예시

### 꼭 강조해야 할 방향
- 배포와 데이터 정합성 전에 AI 확장을 먼저 하지 않도록 유도
- Canonical Form과 데이터 Flow 정합성을 먼저 고정
- 대시보드와 Agentic 데이터 소스 일치 전에는 AI 고도화를 확장하지 않도록 명시
- 실사용 근거가 없는 Context/Memory는 후순위로 둔다

## 2. lessons-learned.md 작성 가이드

### 역할
`lessons-learned.md`는 사후 회고표가 아니라,
유사 프로젝트를 위한 실수 방지 플레이북이어야 한다.

### 각 항목에 반드시 포함할 내용
- 문제
- 왜 발생했는가
- 조기 징후
- 예방 규칙
- 코드/설계 수준 대응
- PR 전 점검법
- 재발 가능 범위
- 관련 blueprint 단계
- 관련 reusable pattern

### 우선순위가 높은 교훈
- 요청 스코프 DB 세션을 백그라운드 태스크에 전달하지 말 것
- `order_by(asc) + limit` 방식으로 최신 데이터를 조회하지 말 것
- `with_structured_output` 의존보다 JSON Mode를 우선 검토할 것
- reasoning 모델 옵션 비호환성을 새 모델 도입 시 바로 점검할 것
- 인메모리 캐시에는 Cache Warmup 전략을 기본 고려할 것
- 배포는 불안정한 빌드 체인보다 Dockerfile 우선 전략을 둘 것
- 대시보드와 Agentic 내부 데이터 소스를 반드시 일치시킬 것

### 기대 결과
이 문서를 읽으면 "이 문제를 겪었다"가 아니라
"이 문제를 다음 프로젝트에서 어떻게 미리 막는가"가 보여야 한다.

## 3. wrapup-summary.md 작성 가이드

### 역할
`wrapup-summary.md`는 프로젝트 자랑 문서가 아니라,
문서 패키지 전체의 입구 역할을 해야 한다.

### 반드시 담아야 할 내용
- 이 패키지로 무엇을 얻을 수 있는가
- 어떤 유형의 프로젝트에 적합한가
- 먼저 읽을 순서
- 가장 중요한 실수 5개
- 가장 바로 재사용할 수 있는 패턴 5개
- 권장 개발 순서 1페이지 요약
- 다른 문서로 넘어가는 안내

### 추가 원칙
- 테스트 수, 기간, phase 수 등 수치는 문서 간 일관되게 유지한다.
- summary는 요약이 아니라 "이 패키지를 어떻게 활용할지"를 안내해야 한다.

## 4. reusable-patterns 작성 가이드

### 역할
`reusable-patterns/`는 예시 코드 모음이 아니라,
유사 프로젝트를 위한 스니펫 자산 라이브러리여야 한다.

### 각 파일 상단에 반드시 있어야 할 정보
- 용도
- 언제 써야 하는가
- 전제조건
- 의존성
- 통합 포인트
- 주의사항
- 원본 출처

### 품질 기준
- 가능하면 필요한 import 포함
- placeholder는 명시적으로 표시
- 어떤 계층/모듈에 들어가는지 설명
- 실패하기 쉬운 포인트를 같이 적음
- 예시인지, 복붙 가능한지, 수정이 필요한지 분류

### 권장 추가 문서
`reusable-patterns/README.md`

포함 내용:
- 패턴 목록
- 패턴별 해결 문제
- 적용 대상 계층
- 복붙 가능 여부
- 수정 필요 여부

## 5. 문서 간 연결 원칙

각 문서는 따로 놀면 안 된다.
서로 연결되어야 유사 프로젝트에서 실제로 쓸 수 있다.

### 연결 방식
- blueprint 단계에서 관련 lesson ID를 참조
- lessons 항목에서 관련 pattern 파일을 참조
- summary에서 각 문서의 역할과 읽는 순서를 안내
- 가능하면 공통 ID 체계를 둔다
  - phase id
  - lesson id
  - pattern id

## 6. 최종 품질 기준

개편된 `project-wrapup`는 아래 조건을 만족해야 한다.

- blueprint만 읽어도 권장 개발 순서를 결정할 수 있어야 한다.
- lessons는 예방 규칙 중심이어야 한다.
- reusable-patterns는 실제 재사용 가능한 자산처럼 보여야 한다.
- summary는 문서 패키지의 입구 역할을 해야 한다.
- 고급 기능은 코어 경로보다 뒤로 밀려야 한다.
- trial-and-error를 줄이기 위한 의사결정 규칙이 분명해야 한다.
- 회고성 문장보다 지침성 문장이 더 많아야 한다.

## 한 줄 기준
이 문서 묶음은 "우리가 뭘 했는가"를 남기는 것이 아니라,
"다음 유사 프로젝트가 무엇을 어떻게 해야 시행착오를 줄일 수 있는가"를 남겨야 한다.

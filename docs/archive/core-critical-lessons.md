## 1. Dashboard 데이터 매니지먼트
- UX와 데이터 Flow 정합성: 모든 사용자 UX 시나리오를 검토할 것. 거기에 맞게 데이터 Flow가 일어나는지 점검할 것. 특히 페이지 간 이동이나 페이지내의 탭간 이동, 그리고 파라미터에 따른 출력 데이터가, 대시보드 전체에서 일관되게 중앙 관리되도록 구성할 것. 이 원칙은 이후 Agentic Flow에도 동일하게 적용한 프레임워크가 되어야 함
- 페이지 책임제: 페이지내의 정보 (내용, 파라미터)는 명확히 범위 및 항목이 정의할 것. 특히 모든 변수를 Canonical Form으로 구성하여, 대시보드 전체에서 (+Agentic Flow에도 연결) 일관되게 재사용할 것

## 2. Agentic Flow 가시성 및 정합성
- 전제: Agentic Flow 복잡성을 가시화하는 것이 문제 해결에 출발점. 개발 단계마다 README.md등을 통해 최신 Agentic Flow를 가시성있게 업데이트
- 가시성 확보 전략: 각 노드 명시. 노드내의 모든 함수 명시. 함수가 재활용되는 모든 관련 노드를 모두 명시하여 영향 범위를 명확화
- 정합성 확보 전략
  - 대시보드 데이터 Flow와 Agentic Flow 정확히 일치. 특히 내부 사용 함수도 1:1 매칭되는지 확인
  - 페이지내 정합성: 대시보드의 페이지 정보 (내용, 파라미터)와 Identical한 정보를 Agentic Flow가 받고 사용자에게 반환할 수 있어야 함 

## 3. E2E 테스트 환경
### 3-1. 기본 전제
- 로컬 서버에서 테스트 한후 프로덕션 배포 테스트를 진행한다
- 로컬 서버 테스트 하는 경우 서버 환경이 문제없이 설정되어 있는지 사전 확인후 테스트 진행. 테스트 오류 발생시 우선적으로 서버 환경의 문제가 없었는지 재확인하고 다른 문제로 넘어가는 디버깅 Sequence를 기본 Rule로 적용할 것
- 로깅 설정: 샌드박스내에서 Python등 언어별 Default 로깅 설정(특히 레벨)을 정확히 확인할 것. WARNING/DEBUG/INFO 설정이 맞게 되었는지 테스트 시작전 재확인


### 3-2. 로컬 서버 환경
- 백엔드 및 프론트 서버 각각 부팅한후 브라우저 테스트 진행
- 코드 변경 사항이 reloading되어 제대로 서버 API에 반영이 되어 있는지 확인 필요
- 코드 수정 및 테스트가 잦은 경우, 서버를 추가로 기동하면서 여러 taskid가 공존하는 문제 발생 -> single server로 유지되도록 테스트마다 재확인 과정
- 터미널 Bash: 브라우저 테스트가 아닌 클로드 코드가 터미널 Bash를 통해 백엔드 response 테스트하는 표준 프레임워크가 있어야 함
- 입출력: 로깅등을 터미널 입출력하거나 파일로 stdout하는 경우 문법에 주의. 필요시 별도 스크립트로 운영하는 방법 고려. 터미널 Bash 항목과 마찬가지로 표준 프레임워크로 수행하여 작업 오류율을 Zero화할 수 있어야 함
